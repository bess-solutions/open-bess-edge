# BESSAI — Patch Management SLA

**Version:** 1.0  
**Date:** 2026-02-22  
**Status:** Active  
**Applies to:** `open-bess-edge` — all production deployments  
**Owner:** BESSAI Core Team

---

## Purpose

This document defines the formal **Patch Management Service Level Agreement (SLA)** for the BESSAI Edge Gateway software. It specifies the maximum time windows for identifying, remediating, and deploying security patches based on vulnerability severity.

This SLA satisfies:
- IEC 62443-3-3 SR 2.2 (patch and software vulnerability management)
- OpenSSF Best Practices — Silver/Gold criteria
- BESSAI IEC 62443 SL-2 certification path requirements

---

## Scope

This SLA covers:
- All code in `src/`, `scripts/`, `infrastructure/` within `open-bess-edge`
- All direct and transitive dependencies listed in `requirements.txt`
- Docker base images referenced in `infrastructure/docker/Dockerfile`

---

## Severity-Based SLA

| CVSS v3.1 Score | Severity | Identification | Patch Available | Deployment to Operators |
|---|---|---|---|---|
| 9.0 – 10.0 | **Critical** | Continuous (CVE feeds + Dependabot) | **14 calendar days** | **3 business days** after patch |
| 7.0 – 8.9 | **High** | Continuous | **30 calendar days** | **7 business days** after patch |
| 4.0 – 6.9 | **Medium** | Weekly scan (CI Fridays) | **90 calendar days** | Next scheduled maintenance window |
| 0.1 – 3.9 | **Low** | Weekly scan | Best effort — next minor release | With release upgrade |

> [!IMPORTANT]
> SLA clock **starts** from the earlier of: (a) first internal awareness, or (b) public CVE publication.

|---|---|---|---|

| External reporter | PSIRT process | Ad-hoc | PSIRT Lead |

## Patch Prioritization Matrix

When multiple vulnerabilities exist simultaneously, prioritize using this matrix:

| Factor | Weight |
|---|---|
| CVSS Base Score | 40% |
| Exploitability (Attack Vector: Network > Local) | 25% |
| Impact on safety functions (`safety.py`, `modbus_driver.py`) | 25% |
| Exposure (internet-facing vs. OT-isolated) | 10% |

**Safety-critical components** (`src/core/safety.py`, `src/drivers/modbus_driver.py`) receive priority escalation: their patch SLA is reduced by **50%** regardless of CVSS score.

## Patch Development Process

1. **Detection** — CVE identified via scan or PSIRT report
2. **Triage** — CVSS score confirmed, components affected identified
3. **Branch** — `security/cve-YYYY-NNNNN` created from `main`
4. **Development** — Patch written; test suite must remain 100% passing
5. **Review** — Two-person review (Core Maintainer, not patch author)
6. **Merge & Tag** — Merged to `main`, new patch release tagged
7. **Advisory** — GitHub Security Advisory published (for CVSS ≥ 4.0)
8. **Notification** — Known production operators notified via email

---

## Operator Responsibilities

Operators deploying BESSAI Edge Gateway in production must:

1. **Subscribe to security advisories:** GitHub → Watch → Security Alerts
2. **Deploy patches within the operator SLA:**
   - Critical/High: within **7 days** of patch release
   - Medium: within **30 days** of patch release
3. **Maintain a supported version:** Patches are only backported to the **current stable minor release**
4. **Report incidents:** Notify BESSAI PSIRT if a vulnerability is being actively exploited in your deployment

---

## SLA Exceptions

Extensions to the standard SLA may be granted in writing by the PSIRT Lead when:

- The fix requires a major refactoring that introduces regression risk
- The vulnerability is in a third-party library with no upstream fix available
- The operator requires an emergency change freeze (e.g., during plant commissioning)

**Maximum extension:** +30 days for Critical, +60 days for High. Extensions are documented in the GitHub Security Advisory.
