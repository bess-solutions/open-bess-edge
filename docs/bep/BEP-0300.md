## Abstract

BEP-0300 define el protocolo para activar el agente DRL (`ONNXArbitrageAgent`) en modo de **control autónomo real** — emitiendo comandos `write_tag()` al hardware BESS. Actualmente el agente opera en modo `observe-only` (BEP-0200 Fase 2). Esta propuesta establece los criterios de seguridad, el proceso de staging validado, los mecanismos de rollback automático y los KPIs de producción requeridos antes de habilitar el control real.

---

## Motivation

El agente DRL en modo observe-only demuestra un +33.5 % de mejora de ingresos vs. la política rule-based (BENCHMARK-004). Sin embargo, el valor económico real solo se materializa cuando el agente controla efectivamente la carga/descarga del BESS. Las barreras para el paso a producción son de **seguridad**, no de desempeño — esta BEP las aborda sistemáticamente.

**Impacto estimado:** Gestión totalmente autónoma de BESS, reducción de intervención humana a supervisión esporádica, habilitación de respuesta de frecuencia en <500 ms sin operador.

## Specification

### Prerequisitos obligatorios (MUST)

Antes de habilitar `BESSAI_DRL_ENABLED=write`, el operador MUST verificar:

1. **Staging Pass**: El agente debe operar en modo `observe-only` durante un mínimo de **7 días continuos** en el sitio de producción, sin divergencias > 5 % entre acción propuesta y acción rule-based.
2. **Safety Guard activo**: `SafetyGuard` (BEP-0200) configurado con límites de hardware certificados por el fabricante.
3. **WatchdogManager activo**: `WatchdogManager` (+BEP-0200 Fase 2) habilitado con backoff exponencial y fallback a rule-based.
4. **Rollback automático probado**: Test de chaos engineering que demuestre rollback en < 200 ms.
5. **Cobertura de tests ≥ 90 %** en `src/agents/drl_agent.py` y `src/core/safety.py`.

### Arquitectura de control (Write Mode)

```
ONNXArbitrageAgent
        │
        ▼ (acción propuesta)
  SafetyGuard.validate(action)   ← MUST: SoC ∈ [10%, 90%], Temp < 45°C
        │ OK
        ▼
  write_tag("target_power_kw", value)
        │
        ▼
  ModbusDriver → Hardware BESS
        │
        ▼ (confirmación)
  WatchdogManager.heartbeat()
```

### Variable de entorno de activación

```bash
# Habilitación escalonada
BESSAI_DRL_ENABLED=observe   # Fase actual — solo logging
BESSAI_DRL_ENABLED=shadow    # NUEVO — escribe pero en registro paralelo, no al hardware
BESSAI_DRL_ENABLED=write     # Producción — control real
```

El modo `shadow` es nuevo: el agente computa la acción y la registra en telemetría, pero no ejecuta `write_tag()`. Permite auditar desviaciones vs. rule-based en tiempo real sin riesgo.

### Métricas de aceptación (MUST cumplir durante staging)

| KPI | Umbral mínimo requerido |
|---|---|
| Mejora de ingresos vs. rule-based | ≥ +15 % (sobre 30 días) |
| Violaciones de `SafetyGuard` | 0 (cero absoluto) |
| Latencia de inferencia ONNX | < 10 ms (p99) |
| Tiempo de rollback automático | < 200 ms |
| Disponibilidad del agente | ≥ 99.9 % uptime |

### Rollback automático

```python
# src/core/main.py — Step 5e (write mode)
try:
    action = drl_agent.act(observation)
    safety_guard.validate(action)       # raises SafetyViolationError si fuera de rango
    write_tag("target_power_kw", action)
    watchdog.heartbeat()
except (SafetyViolationError, ModbusException) as e:
    logger.critical("DRL_ROLLBACK", reason=str(e))
    rule_based_policy.act(observation)  # Fallback inmediato
    metrics.drl_rollback_total.inc()
```

## Backward Compatibility

- **Totalmente retrocompatible** — el modo `observe` actual (BEP-0200 Fase 2) no es afectado.
- `BESSAI_DRL_ENABLED` sigue siendo `observe` por defecto. El operador debe optar explícitamente por `write`.
- El `ArbitragePolicy` rule-based existente se mantiene como fallback permanente.

## Rejected Alternatives

| Alternativa | Razón de rechazo |
|---|---|
| Activar write directamente sin staging | Riesgo operacional inaceptable — sin evidencia de comportamiento en-site |
| Control proporcional (write parcial) | Ambigüedad en la responsabilidad de control — mejor control total con rollback |
| MPC (Model Predictive Control) como alternativa a DRL | Mayor complejidad computacional; DRL ya demuestra +33.5% en benchmark |
