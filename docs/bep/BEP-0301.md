## Abstract

BEP-0301 especifica la integración de una **capa de explicabilidad (XAI)** en los agentes de IA de BESSAI. Se utiliza SHAP (SHapley Additive exPlanations) para generar explicaciones de las decisiones del `ONNXArbitrageAgent` y del `ModbusAnomalyDetector` (AI-IDS). Las explicaciones se publican como telemetría estructurada (Prometheus + OpenTelemetry), habilitando auditoría en tiempo real por operadores, reguladores y adopters.

---

## Motivation

A medida que BESSAI avanza hacia control autónomo (BEP-0300), la capacidad de **explicar** cada decisión se vuelve crítica por tres razones:

1. **Confianza del operador**: Un operador que no entiende *por qué* el agente ordenó una descarga en un momento dado no aprobará el modo write.
2. **Cumplimiento regulatorio**: EU AI Act (categoría "High-Risk AI") y NTSyCS Chile exigen trazabilidad de decisiones en sistemas de control de infraestructura crítica.
3. **Debugging y mejora continua**: SHAP permite identificar qué variables tienen mayor peso en decisiones erróneas, acelerando el ajuste del modelo.

## Specification

### Componentes

#### 1. `src/interfaces/xai_explainer.py` (NUEVO)

```python
import shap
import numpy as np
from typing import Any

class BESSAIExplainer:
    """SHAP-based explainer for BESSAI AI agents."""

    def __init__(self, model_fn, background_data: np.ndarray, feature_names: list[str]):
        self.explainer = shap.KernelExplainer(model_fn, background_data)
        self.feature_names = feature_names

    def explain(self, observation: np.ndarray) -> dict[str, float]:
        """Returns SHAP values as {feature_name: shap_value} dict."""
        shap_values = self.explainer.shap_values(observation, nsamples=50)
        return dict(zip(self.feature_names, shap_values[0]))

    def top_factors(self, observation: np.ndarray, n: int = 3) -> list[tuple[str, float]]:
        """Returns top-N most influential features for this decision."""
        shap_dict = self.explain(observation)
        return sorted(shap_dict.items(), key=lambda x: abs(x[1]), reverse=True)[:n]
```

#### 2. Integración en `main.py` Step 5e

```python
# Tras cada decisión del agente DRL
explanation = xai_explainer.top_factors(observation)
structlog.get_logger().info(
    "drl_decision_explained",
    action=action,
    top_factors=explanation,
    site_id=settings.site_id,

# Publicar en OpenTelemetry como span attribute
with tracer.start_as_current_span("drl_decision") as span:
    span.set_attribute("xai.top_factor_1", explanation[0][0])
    span.set_attribute("xai.shap_value_1", explanation[0][1])
```

### Requisitos normativos (MUST)

- Toda decisión del `ONNXArbitrageAgent` en modo `write` MUST generar una explicación SHAP registrada en el log estructurado.
- Las explicaciones MUST ser retrievable vía el endpoint `/api/v1/explain/last` del `DashboardAPI`.
- El sistema MUST poder generar un **reporte de auditoría XAI** (JSON) para los últimos N ciclos de decisión.

## Backward Compatibility

- Las explicaciones son **adicionales** — no modifican el comportamiento del agente.
- El endpoint `/api/v1/explain/last` es nuevo — no rompe contratos existentes.
- Requiere añadir `shap` a `requirements.txt`. No introduce dependencias incompatibles.

## Rejected Alternatives

| Alternativa | Razón de rechazo |
|---|---|
| LIME (Local Interpretable Model-agnostic Explanations) | SHAP tiene propiedades teóricas más sólidas (valor de Shapley) y mejor soporte para modelos ONNX |
| Integrated Gradients | Requiere acceso al grafo del modelo — incompatible con despliegue ONNX en edge |
| Explicaciones off-line (batch) | No útil para auditoría en tiempo real que requieren operadores y reguladores |
