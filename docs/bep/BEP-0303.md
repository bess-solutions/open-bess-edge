## Abstract

BEP-0303 especifica **BESSAIEvolve** — un motor de mejora continua autónoma para los agentes de IA de BESSAI, inspirado en los principios de AlphaEvolve (DeepMind, 2025). El sistema genera periódicamente una población de candidatos de política (mutando parámetros de reward shaping y thresholds de ArbitragePolicy), los evalúa en un sandbox con datos CMg históricos, selecciona al ganador por torneo, y abre un Pull Request automático con la política mejorada. El ciclo corre cada lunes vía GitHub Actions, sin intervención humana en el bucle evolutivo — solo revisión del PR final.

---

## Motivation

Los principios de AlphaEvolve (DeepMind) demuestran que los programas pueden mejorarse automáticamente mediante:
1. **LLM como mutador** — propone variaciones estructuradas de código/parámetros
2. **Evaluación automatizada** — función de fitness objetiva sin sesgo humano
3. **Ciclo evolutivo continuo** — progreso acumulativo convergente
4. **Diversidad de población** — evita mínimos locales

BESSAI ya tiene todos los ingredientes: `BESSArbitrageEnv` (fitness evaluator), `ArbitragePolicy` (espacio de mutación), `SimulatorDriver` (sandbox), y un workflow semanal (trigger). BESSAIEvolve conecta las piezas.

**Impacto proyectado:**
- +5–15 % adicional de ingresos por año sobre la política estática actual
- Adaptación automática a cambios estructurales del mercado CEN (nuevas renovables, cambio de patrones CMg)
- Convergencia documentada y trazable vía historial de fitness en `models/evolution/`

### Espacio de mutación (Cromosoma)

Un candidato es un `dict` de parámetros que configura `ArbitragePolicy` + reward shaping de `BESSArbitrageEnv`:

```python
@dataclass
class PolicyCandidate:
    id: str                          # UUID4
    generation: int
    parent_id: str | None            # trazabilidad
    params: dict[str, float]         # el "cromosoma"
    fitness: float | None = None     # se asigna tras evaluación

# Espacio de búsqueda
PARAM_SPACE = {
    # ArbitragePolicy thresholds (USD/MWh norm)
    "cmg_low_threshold_norm":  (0.05, 0.20),   # baseline: 30/300 = 0.10
    "cmg_high_threshold_norm": (0.15, 0.40),   # baseline: 80/300 = 0.267
    "soc_min":                 (0.10, 0.25),   # baseline: 0.15
    "soc_max":                 (0.85, 0.98),   # baseline: 0.95
    # Reward shaping (BESSArbitrageEnv)
    "noise_std":               (0.5, 5.0),     # CMg noise (USD/MWh)
    "battery_cost_usd_kwh":   (150.0, 400.0), # degradation penalty weight
}
```

### Algoritmo evolutivo (µ + λ evolution strategy)

```
Generación 0: población inicial (N=10) — baseline + 9 mutantes aleatorios
     │
     ▼ Evaluación paralela en sandbox (ThreadPoolExecutor)
     │
     ▼ Tournament selection: top 3 → padres
     │
     ▼ Generate offspring (λ=7) via Gaussian mutation σ=0.1
     │
     ▼ Elitism: mantener el mejor de la generación anterior
     │
     ▼ Nueva generación (N=10)
     │
     └── Repetir N_GENERATIONS=5 (=50 evaluaciones totales por run semanal)
```

### Operador de mutación (sin LLM externo en v1)

```python
def mutate(parent_params: dict, sigma: float = 0.1) -> dict:
    """Gaussian perturbation — análogo a la mutación en AlphaEvolve."""
    child = {}
    for key, val in parent_params.items():
        lo, hi = PARAM_SPACE[key]
        noise = random.gauss(0, sigma) * (hi - lo)
        child[key] = max(lo, min(hi, val + noise))
    return child
```

> En v2, el operador de mutación llamará a Gemini API para proponer variaciones semánticamente guiadas del reward shaping.

### Criterio de promoción

Un candidato ganador se promueve si:
1. `fitness > baseline_fitness × 1.05` (mejora ≥ 5 % sobre baseline)
2. Cero violaciones de `SafetyGuard` durante evaluación
3. `fitness > best_ever_fitness` (nuevo récord histórico)

Si se cumplen las 3 condiciones → el workflow abre un PR automático con `models/evolution/best_candidate.json`.

### Garantías de seguridad

- La evolución ocurre **exclusivamente en sandbox** — nunca toca hardware real
- El modelo ONNX de producción **no se reemplaza automáticamente** — requiere merge del PR
- `SafetyGuard` se ejecuta en todos los pasos de evaluación
- Si la evaluación falla (crash, timeout) → se descarta el candidato, se continúa

## Rejected Alternatives

| Alternativa | Razón de rechazo |
|---|---|
| Bayesian Optimization (Optuna) | No incorpora diversidad de población ni operadores evolutivos — converge más rápido pero a mínimos locales |
| Llamada directa a LLM en v1 | Añade dependencia externa y latencia; la mutación gaussiana es suficiente para convergencia inicial |
| Reemplazar ONNX automáticamente sin PR | Riesgo operacional: cambios de política sin revisión humana no aceptables en infraestructura crítica |
